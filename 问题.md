   - `L_mask_render_full` 和 `L_sem_proj_2d` 只在 valid 区域计算

### 2.2 模型主路径

当前总模型入口是 [architecture.py](/lustre/fs12/portfolios/nvr/projects/nvr_lpr_nvgptvision/users/shihaow/driving_data/DVGT/dvgt_occ/models/architecture.py)。

主路径是：

1. `DVGT cached tokens -> TokenReassembly`
2. `DynamicDenseBranch`
3. `OccHead`
4. `DynamicQueryDecoder`
5. `GSHead`
6. `GaussianMultiViewMerge`
7. `GaussianQueryAssignment`
8. `GSOccLocalBridge + GSOccGlobalLatentBridge`
9. `EntityAggregator`
10. `OccSemanticProjector`
11. `GaussianMaskRenderer`

### 2.3 当前 GS 实现方式

当前 GS 不是完整成熟的 4DGS 实现，而是“可训练骨架 + 近似 renderer”。

#### GSHead

当前 [gs_head.py](/lustre/fs12/portfolios/nvr/projects/nvr_lpr_nvgptvision/users/shihaow/driving_data/DVGT/dvgt_occ/models/gaussian/gs_head.py) 的形式：

- 输入：reassembled 2D feature + dynamic dense logits + `xyz_1_8`
- 在 `1/8` 分辨率上为每个 `T x V x H/8 x W/8` 位置预测一组 Gaussian candidate
- 输出字段：
  - `center`
  - `offset`
  - `opacity`
  - `scale`
  - `rotation`
  - `feat_dc`
  - `keep_score`
  - `instance_affinity`
  - `motion_code`

也就是说，当前 GS 候选是 **per-view dense grid candidates**，不是先验对象级实体。

#### 跨 view 合并

当前通过 `GaussianMultiViewMerge` 做一次跨 view 去重/合并，然后进入后续 assignment / entity aggregation。

#### Query -> Gaussian

当前通过 `GaussianQueryAssignment` 做 query 到 Gaussian 的分配，并带 `background/null` 分支。

### 2.4 当前 renderer 实现方式

当前 [gaussian_mask_renderer.py](/lustre/fs12/portfolios/nvr/projects/nvr_lpr_nvgptvision/users/shihaow/driving_data/DVGT/dvgt_occ/models/rendering/gaussian_mask_renderer.py) 已经是“camera-aware differentiable renderer”，但它仍然是非常简化的版本。

当前实现特点：

1. 先把 Gaussian `center` 从 `ego0` 变回 world，再通过 `camera_to_world` 投到相机
2. 用 `scale.mean()` 得到单标量 sigma
3. 对每个 Gaussian 只向 `2x2` 邻域像素做 splat
4. 使用简单高斯核权重
5. 没有显式深度排序 / alpha compositing 体渲染
6. 颜色目前只来自 `feat_dc`
7. 最终输出：
   - `render_rgb_all`
   - `render_rgb_dynamic`
   - `render_rgb_static`
   - `render_alpha_all`
   - `render_alpha_dynamic`
   - `render_alpha_static`
   - `sem_proj_2d`

因此，当前 renderer 更接近“可微近似投影器”，还不是成熟 3DGS 系统里那种高保真 image formation。

### 2.5 当前 loss 合同

当前 [loss_builder.py](/lustre/fs12/portfolios/nvr/projects/nvr_lpr_nvgptvision/users/shihaow/driving_data/DVGT/dvgt_occ/training/loss_builder.py) 中：

- `L_gs_render`
  - 主体是 masked `L1`
  - 若 `lpips` 可用，会逐步加入一个很小权重的 LPIPS
  - 否则退化为 multiscale L1
- `L_mask_render_full`
  - 对 `render_alpha_dynamic` 和 `render_alpha_all` 分别做
  - `balanced BCE + soft Dice`
- `L_occ / L_sem_occ / L_dyn_occ`
  - 正常存在
- `L_query2gs`
  - 已启用
- `L_occ_local2gs / L_gs2occ_local / L_sem_proj_2d / L_motion_cons / L_gs_sparse`
  - 当前 run 中 raw loss 会计算，但权重是 `0`

这意味着：

- 当前虽然已经有 `photometric` 分支，但 GS 训练仍然高度依赖 proxy + auxiliary 路线
- 当前 run 还没有真正启用 bridge / sem-proj / motion consistency 的联合约束

## 4. 当前最值得怀疑的问题

下面这些问题，不是“可能有一点点不完美”，而是我认为当前最需要专家帮助判断的重点。

### 问题 1：`query2gs` 基本没学动

现象：

- 训练和验证里的 `loss_query2gs` 都长期钉在 `13.6 ~ 13.8`
- 200 step 内没有看到明显下降

这通常意味着至少以下几种可能：

1. `GaussianQueryAssignment` 的分配目标太弱或太稀疏
2. `query` 表征和 `gaussian instance/motion` 表征根本不在一个可对齐空间
3. `query2gs` 的 loss 形式、温度、负样本设计或 null/background 路由不合理
4. 当前 GS 候选本身质量太差，导致 query 没东西可对齐

这是我目前最明确的主问题。

### 问题 2：GS photometric 分支几乎没动

现象：

- `val_render_psnr` 基本固定在 `1.954`
- `val_render_l1` 基本固定在 `1.2447`
- `loss_gs_render` 也没有稳定下降趋势

这说明：

- 虽然训练图中已经存在 `render_rgb_all -> rgb_target` 的 photometric loss
- 但当前这条分支没有表现出真正的重建学习

可能原因：

1. 当前 renderer 过于简化，不足以形成有效图像监督
2. 当前 per-view dense candidate 的高斯参数化不适合 photometric learning
3. 颜色只用 `feat_dc`，表达能力太弱
4. 没有显式深度排序 / proper compositing，render 信号失真太大
5. 真实 driving scene 对当前 2x2 splat 近似来说过难

### 问题 3：Occ 在学，但仍明显偏弱

现象：

- `val_occ_iou` 从 `0` 提升到 `0.005`
- 不是完全没学，但值依旧很低

同时看分布：

- `val_occ_target_pos_rate ≈ 0.03477`
- `val_occ_pred_pos_rate` 从 `0` 到 `0.00299`
- 仍然远低于 target positive rate

这更像是：

- 模型正在从“全空”往“开始点亮一点 occupancy”走
- 但预测仍然明显偏稀

我目前不敢说这是异常 bug，也可能只是训练阶段太早，但它确实说明：

- **现在还不能说 occupancy 主任务已经起来**

### 问题 4：dynamic mask 的当前 val 基本没有信息量

现象：

- `val_dyn_mask_iou = 0.0`
- 但 `val_dyn_mask_target_pos_rate = 0.0`

这说明当前这 4 个 val batch 里根本没有有效动态正样本。

结果就是：

- `dyn_mask_iou = 0` 不能说明模型真的差
- 也不能说明模型真的好
- 这个指标当前批次上几乎没有判别力

可能原因：

1. `val_batches=4` 太少
2. 当前 val scene / clip 采样刚好非常空
3. `sam3_valid_mask_full` gating 太严格，把正样本都裁掉了
4. 当前 full-res dynamic supervision 的路由仍有口径问题

### 问题 5：bridge 的 raw loss 在上涨，后续可能一开权重就炸

现象：

- `loss_gs2occ_local` raw value 从很小涨到 `0.35`
- `val_loss_gs2occ_local` 到 `step 199` 已经到 `0.424`

虽然当前权重是 `0`，暂时不影响总训练，但这说明：

- 这一支并没有自然收敛
- 后续如果直接按计划把它打开，很可能 destabilize 主训练

换句话说：

- 当前“Stage C 能跑”不等于“Stage D 开桥接后还稳”

### 问题 6：当前 GS 表示形式可能本身就太弱

当前 GS 不是“少量高质量 scene/entity gaussians”，而是：

- 每个 `view / 1/8 grid` 都预测一个 Gaussian candidate
- 然后依赖 merge / assignment / entity aggregator 后处理

这种设计可能的问题：

1. 候选太密、太局部，导致 query 很难形成清晰对象归属
2. per-view dense candidates 对 photometric 监督不友好
3. 跨 view merge 之前，高斯语义仍然偏“2D特征点”，而不是“3D实体”

也就是说，问题可能不只是 loss，而是 **GS 的基本承载形式还不对**。

## 5. 哪些现象我认为“不一定是 bug”

为了避免误判，这里把我认为“目前还不能直接判死刑”的现象列出来。

### 5.1 Occ IoU 很低，不一定就是实现错

原因：

- occupancy 任务本身稀疏
- 当前只训了 200 step
- 从 `pred_pos_rate = 0` 到 `0.003`，至少说明不是完全不学

所以：

- `occ_iou` 当前很差
- 但不能仅凭这一点就断言代码实现一定错误

### 5.2 train loss 在降，不等于主目标已经成立

这是当前最需要警惕的误读。

当前：

- `loss_total` 在降
- `occ / sem_occ / dyn_occ` 在降

但同时：

- `query2gs` 不动
- `render_psnr` 不动
- `dyn mask val` 没信息量

所以不能因为总 loss 在降，就把当前系统当作“已经 ready for scale”。

1. **现在的问题不是单点 bug，也不是“多训一会儿自然就会好”。**
2. **当前 200-step pilot 仍处在 warmup 早期，不能直接拿来判系统最终上限。**
3. **Occ 分支有早期学习迹象，但 entity-aware GS 主链还没有真正 work。**

- 但如果某些分支完全横盘，而另一些分支在动，那就说明不只是训练太短，而是**训练太短 + 主链实现偏弱**叠在一起
